## 三、各模块详细设计

### 3.1 模型选择：RetinexFormer (ICCV 2023)

#### 为什么选择 RetinexFormer

```
对比其他方案：

方案A：从零训练 U-Net + RepVGG
├─ 需要 200 epochs 训练
├─ 需要合成暗光数据
├─ 风险：训练效果不确定
└─ 优势：展示网络设计能力

方案B：使用 RetinexFormer 预训练模型 ✅ 选择
├─ 无需训练，直接使用
├─ SOTA 效果 (ICCV 2023)
├─ 物理可解释 (Retinex 分解)
└─ 优势：专注工程部署，面试可讲原理
```

#### RetinexFormer 架构

```
输入图像 I (低光)
       │
       ▼
┌─────────────────────────────────┐
│  Illumination Estimator (IE)    │  估计光照分量 L
│  - Multi-scale Transformer      │
│  - 全局注意力捕获光照分布        │
└─────────────────────────────────┘
       │
       ▼
┌─────────────────────────────────┐
│  Illumination-Guided Attention  │  光照引导增强
│  (IGA Transformer Block)        │
│  - 根据光照自适应增强           │
└─────────────────────────────────┘
       │
       ▼
┌─────────────────────────────────┐
│  Reflectance Restoration        │  恢复反射率 R
│  - CNN Decoder                  │
│  - 细节恢复和降噪               │
└─────────────────────────────────┘
       │
       ▼
输出 = R * enhanced_L (Retinex 重建)

核心原理：Retinex 理论
I = L × R
├─ I: 观察到的图像
├─ L: 光照分量 (Illumination)
└─ R: 反射率 (Reflectance，物体本身颜色)

低光增强 = 提升光照 L，保持反射率 R
```

#### 模型规格

| 属性 | 值 |
|------|-----|
| 参数量 | ~3-4M |
| 输入 | RGB, [0, 1] 归一化 |
| 输出 | RGB, [0, 1] 增强图像 |

| PSNR | ~22-24 dB |

### 3.2 Tiling 分块处理（工程层核心）

#### 为什么需要 Tiling

```
问题：手机拍照 12MP (4000x3000)，直接推理会 OOM
方案：切成小块分别处理，再拼回去

Tile 大小选择：
├─ 太小 (128x128)：块太多，开销大，边界问题多
├─ 太大 (1024x1024)：单块内存占用高
└─ 推荐：512x512，平衡内存和效率
```

#### Overlap 设计

```
为什么要 Overlap：
├─ 神经网络在图像边缘的输出质量较差
├─ 直接拼接会有明显接缝
└─ Overlap 区域做融合可以消除接缝

┌─────────────────┬─────────────────┐
│     Tile 0      │overlap│  Tile 1 │
│                 │  32px │         │
└─────────────────┴───────┴─────────┘

Overlap 大小：32 像素（可调）
```

#### 融合策略

```cpp
// 线性加权融合
// 在 overlap 区域，左边 tile 权重从 1 递减到 0
// 右边 tile 权重从 0 递增到 1

for (int i = 0; i < overlap_width; i++) {
    float alpha = (float)i / overlap_width;
    output[x] = left_tile[x] * (1 - alpha) + right_tile[x] * alpha;
}
```

#### 代码结构

```cpp
class TilingProcessor {
public:
    struct Tile {
        int x, y;           // 在原图中的位置
        int width, height;  // tile 尺寸
        cv::Mat data;       // 图像数据
    };

    // 切分大图为多个 tile
    std::vector<Tile> split(const cv::Mat& input,
                            int tile_size,
                            int overlap);

    // 合并处理后的 tiles
    cv::Mat merge(const std::vector<Tile>& tiles,
                  int original_width,
                  int original_height,
                  int overlap);
};
```

### 3.3 线程池并行推理（工程层核心）

#### 设计思路

```
为什么不用 std::async：
├─ 每次调用可能创建新线程，开销大
├─ 无法控制并发数量
└─ 面试官会问"为什么不自己实现"

线程池优势：
├─ 线程复用，避免频繁创建销毁
├─ 可控制并发数（匹配 CPU 核心数）
└─ 任务队列解耦生产者和消费者
```

#### 参考实现

```
推荐参考：progschj/ThreadPool (GitHub)
├─ 单头文件，约 100 行
├─ 使用 C++11 标准库
├─ 代码清晰，适合学习

核心组件：
├─ std::vector<std::thread> workers    // 工作线程
├─ std::queue<std::function<void()>> tasks  // 任务队列
├─ std::mutex queue_mutex              // 保护队列
├─ std::condition_variable condition   // 线程同步
└─ std::atomic<bool> stop              // 停止标志
```

#### 关键代码

```cpp
// 1. 工作线程的循环逻辑
void worker_thread() {
    while (true) {
        std::function<void()> task;
        {
            std::unique_lock<std::mutex> lock(queue_mutex);
            // 等待任务或停止信号
            condition.wait(lock, [this] {
                return stop || !tasks.empty();
            });
            if (stop && tasks.empty()) return;
            task = std::move(tasks.front());
            tasks.pop();
        }
        task();  // 执行任务
    }
}

// 2. 提交任务
template<class F>
auto enqueue(F&& f) -> std::future<decltype(f())> {
    auto task = std::make_shared<std::packaged_task<decltype(f())()>>(
        std::forward<F>(f)
    );
    std::future<decltype(f())> res = task->get_future();
    {
        std::lock_guard<std::mutex> lock(queue_mutex);
        tasks.emplace([task]() { (*task)(); });
    }
    condition.notify_one();
    return res;
}
```

#### 与 Tiling 结合

```cpp
void process_large_image(const cv::Mat& input) {
    // 1. 切分
    auto tiles = tiling.split(input, 512, 32);

    // 2. 并行推理
    std::vector<std::future<cv::Mat>> futures;
    for (auto& tile : tiles) {
        futures.push_back(pool.enqueue([&, tile]() {
            return ncnn_inference(tile.data);  // NCNN 推理
        }));
    }

    // 3. 等待所有任务完成
    for (int i = 0; i < tiles.size(); i++) {
        tiles[i].data = futures[i].get();
    }

    // 4. 合并
    cv::Mat output = tiling.merge(tiles, input.cols, input.rows, 32);
}
```

### 3.4 NCNN 推理与量化（工程层）

#### NCNN 基础推理

```cpp
class NCNNInference {
private:
    ncnn::Net net;

public:
    bool load(const std::string& param_path,
              const std::string& bin_path) {
        net.opt.use_vulkan_compute = false;  // CPU 推理
        net.opt.num_threads = 1;  // 单线程（外层已有线程池）
        net.load_param(param_path.c_str());
        net.load_model(bin_path.c_str());
        return true;
    }

    cv::Mat inference(const cv::Mat& input) {
        // 预处理：BGR → RGB，归一化
        ncnn::Mat in = ncnn::Mat::from_pixels(
            input.data, ncnn::Mat::PIXEL_BGR2RGB,
            input.cols, input.rows
        );
        in.substract_mean_normalize(mean_vals, norm_vals);

        // 推理
        ncnn::Extractor ex = net.create_extractor();
        ex.input("input", in);
        ncnn::Mat out;
        ex.extract("output", out);

        // 后处理：转回 cv::Mat
        cv::Mat result = ncnn_mat_to_cv_mat(out);
        return result;
    }
};
```

#### 模型转换流程

```bash
# 1. 简化 ONNX（必须，否则 NCNN 可能转换失败）
python -m onnxsim model.onnx model_sim.onnx

# 2. 转换为 NCNN
./onnx2ncnn model_sim.onnx model.param model.bin

# 3. 优化（FP16 存储，减小模型体积）
./ncnnoptimize model.param model.bin model_opt.param model_opt.bin 65536
```

#### 量化（INT8）

```bash
# 1. 准备校准数据集（100-500 张代表性图片）
# 2. 生成量化表
./ncnn2table model_opt.param model_opt.bin calibration_images/ model.table

# 3. 量化
./ncnn2int8 model_opt.param model_opt.bin model_int8.param model_int8.bin model.table
```

#### 量化注意事项

```
可能遇到的问题：
├─ 某些层量化后精度下降严重
├─ 颜色出现偏移
└─ 边缘出现伪影

解决方案：
├─ 逐层分析，找出敏感层
├─ 敏感层保持 FP16，其他层 INT8（混合精度）
└─ 增加校准数据的多样性
```

#### BLC 黑电平校正

```cpp
// 参考: references/HDR-ISP-main/srcs/sources/modules/blc.cpp
// 原理：传感器在无光时也会有电信号（暗电流），需要减去

void black_level_correction(cv::Mat& raw, int black_level) {
    raw.forEach<uint16_t>([black_level](uint16_t& pixel, const int*) {
        pixel = (pixel > black_level) ? (pixel - black_level) : 0;
    });
}
```

#### AWB 自动白平衡

```cpp
// 参考: docs/03_AWB模块.md - 灰度世界法
// 原理：假设场景平均颜色为灰色，据此计算RGB通道增益

struct WBGain { float r, g, b; };

WBGain gray_world_awb(const cv::Mat& img) {
    cv::Scalar mean = cv::mean(img);
    float avg = (mean[0] + mean[1] + mean[2]) / 3.0f;

    return {
        avg / mean[2],  // R gain
        avg / mean[1],  // G gain
        avg / mean[0]   // B gain
    };
}

void apply_wb_gain(cv::Mat& img, const WBGain& gain) {
    std::vector<cv::Mat> channels;
    cv::split(img, channels);
    channels[2] *= gain.r;  // R
    channels[1] *= gain.g;  // G
    channels[0] *= gain.b;  // B
    cv::merge(channels, img);
}
```

#### Demosaic 去马赛克

// 参考: references/HDR-ISP-main/srcs/sources/modules/demoasic.cpp

// 简化版：使用 OpenCV 内置函数
cv::Mat demosaic(const cv::Mat& bayer, int pattern = cv::COLOR_BayerRG2RGB) {
    cv::Mat rgb;
    cv::cvtColor(bayer, rgb, pattern);

}

## 六、面试准备要点

### 6.1 必须能回答的问题

#### 模型相关

```
Q: 为什么选择 RetinexFormer？
A: RetinexFormer 基于 Retinex 理论，把图像分解为光照和反射率，符合物理规律。
   Transformer 的全局注意力能捕获大范围光照信息，比纯 CNN 更适合低光增强。
   ICCV 2023 发表，是当前 SOTA 效果。

Q: Retinex 理论是什么？
A: I = L × R，图像 = 光照 × 反射率。低光增强的核心是提升光照 L，
   同时保持物体本身颜色 R 不变。这比直接端到端回归更可解释。

Q: PSNR 和 SSIM 分别衡量什么？
A: PSNR 衡量像素级误差，SSIM 衡量结构相似性，更符合人眼感知。
```

#### 工程相关

```
Q: 为什么要用 Tiling？
A: 12MP 图片直接推理会 OOM。Tiling 把大图切成小块，降低单次内存占用，
   还能利用多核并行加速。

Q: Overlap 区域怎么处理？
A: 用线性加权融合。在 overlap 区域，左边 tile 权重从 1 递减到 0，
   右边从 0 递增到 1，实现平滑过渡，消除接缝。

Q: 为什么不用 std::async？
A: std::async 每次可能创建新线程，开销大且并发数不可控。
   手写线程池可以复用线程，控制并发数匹配 CPU 核心数。

Q: 线程池里 condition_variable 的作用？
A: 让工作线程在没有任务时休眠，有新任务时被唤醒，避免忙等浪费 CPU。
```

#### 部署相关

```
Q: INT8 量化的原理？
A: 把 FP32 权重和激活值映射到 INT8 范围，用整数运算替代浮点运算，
   速度更快，但需要校准数据确定映射的 scale 和 zero_point。

Q: 量化后精度下降怎么办？
A: 逐层分析敏感度，敏感层（通常是首尾层）保持 FP16，其他层 INT8，
   混合精度可以在速度和精度间取得平衡。

Q: 为什么用 NCNN 不用 TensorRT？
A: 手机是 ARM 架构，没有 NVIDIA GPU。NCNN 专为移动端优化，
   支持 ARM NEON 指令集，TensorRT 只能在 NVIDIA GPU 上用。
```

### 6.2 准备的"踩坑"故事

```
故事 1：Tiling 接缝问题
"一开始直接拼接，有明显接缝。分析发现是网络边缘输出不稳定，
 加了 32 像素 overlap 和线性融合后解决。"

故事 2：量化颜色偏移
"INT8 量化后图片偏绿。定位到是第一个卷积层对量化敏感，
 改成 FP16 后颜色恢复正常。"

故事 3：多线程没加速
"加了线程池反而更慢。分析发现 tile 太小导致任务数太多，
 锁竞争严重。增大 tile size 后加速比从 0.8x 提升到 3.2x。"

故事 4：ONNX 转换失败
"NCNN 转换时报算子不支持。检查发现是某个 PyTorch 操作导出了
 不常见算子，换成等效的常见操作后解决。"
```

## 八、项目总结
